## 2. Постановка задачи и контекст

### 2.1 Предметная область и задача

Проект относится к инженерному анализу энергоэффективности зданий. Цель — предсказать:
- Y1: Heating Load (нагрузка на отопление);
- Y2: Cooling Load (нагрузка на охлаждение).

Потенциальный пользователь — инженер-проектировщик или архитектор, который оценивает энергетические характеристики здания на этапе концепции и выбирает параметры конструкции.

### 2.2 Формулировка задачи в терминах ML

- Тип задачи: регрессия.
- Входные данные: 8 признаков здания (`X1..X8`), включающих компактность, площадь, высоту, ориентацию и остекление.
- Целевые переменные: два канала прогноза (`Y1` и `Y2`).
- Ограничения: высокая точность предсказания и воспроизводимость; требования к простому рест API и конфигурируемости через YAML.

### 2.3 Целевые метрики качества

- Основные метрики: R², RMSE, MAE.
- Почему эти метрики:
  - R² показывает долю объяснённой дисперсии и хорошо подходит для сравнения регрессионных моделей;
  - RMSE чувствителен к большим ошибкам и удобен для инженерных задач, где важно ограничить пиковую неточность;
  - MAE показывает среднюю абсолютную ошибку, легче интерпретируется в тех же единицах, что и целевая нагрузка.

## 3. Данные

### 3.1 Источник данных

Данные берутся из открытого набора UCI Energy Efficiency Dataset. Файл размещён в `data/ENB2012_data.csv`.

### 3.2 Структура данных

- Признаки нагрузок:
  - `X1` — Relative Compactness;
  - `X2` — Surface Area;
  - `X3` — Wall Area;
  - `X4` — Roof Area;
  - `X5` — Overall Height;
  - `X6` — Orientation;
  - `X7` — Glazing Area;
  - `X8` — Glazing Area Distribution.
- Цели:
  - `Y1` — Heating Load;
  - `Y2` — Cooling Load.

Типы данных:
- числовые: `X1`, `X2`, `X3`, `X4`, `X5`, `X7`;
- категориальные: `X6`, `X8`.

### 3.3 Предобработка и EDA

Основной пайплайн предобработки описан в `src/data/loader.py` и `src/features/preprocess.py`:
- загрузка CSV с разделителем `;` и десятичным `,`;
- удаление строк с пропусками (`dropna(how='all')`);
- приведение категориальных признаков `X6` и `X8` к типу `category`;
- разбиение на `train/test` с `test_size=0.2`, `random_state=42`.

Препроцессор:
- `StandardScaler` для числовых признаков;
- `OneHotEncoder(handle_unknown='ignore')` для категориальных.

EDA выполнялся в ноутбуке `notebooks/exp01_eda.ipynb`, где строились:
- матрица корреляций;
- распределения целевых переменных;
- зависимость `Y1` и `Y2` от ключевых признаков;
- анализ `X6`/`X8` и влияния остекления.

## 4. Модели и подходы

### 4.1 Базовые модели

В проекте сравниваются четыре класса моделей:
- `LinearRegression` — простой baseline;
- `RandomForest` — ансамбль деревьев;
- `CatBoost` — градиентный бустинг для числовых и категориальных признаков;
- `Stacking` — финальный ансамбль на основе нескольких моделей.

Конфигурации заданы в `configs/experiment.yaml`.

### 4.2 Улучшенные модели и эксперименты

Финальный подход основан на стекинге, где в качестве базовых моделей используются:
- `CatBoost`;
- `RandomForest`;
- `LinearRegression`;
- итоговый мета-алгоритм — `LinearRegression`.

Важные настройки:
- `stacking.cv = 5`;
- `n_jobs=-1` для использования всех доступных ядер;
- `CatBoost` и `RandomForest` настроены на скорость и устойчивость за счёт среднего количества деревьев и выключенного verbose.

### 4.3 Почему выбран Stacking

Stacking показал наибольшую среднюю R² на валидации и тесте, сохраняя стабильность и низкий RMSE. Он комбинирует сильные стороны бустинга и леса, снижая смещение и дисперсию.

## 5. Экспериментальный протокол и результаты

### 5.1 Протокол эксперимента

- Разбиение данных: `train/test = 80/20`, `random_state=42`.
- Предобработка: стандартирование числовых признаков и one-hot кодирование категорий.
- Оценка метрик: на train, valid и test.
- Критерий выбора: `max_mean_R2_valid`.

### 5.2 Сравнение моделей

| Модель | Описание | R² valid | R² test | RMSE test (Y1/Y2) | MAE test (Y1/Y2) |
|---|---|---|---|---|---|
| LinearRegression | baseline | 0.9347 | 0.9020 | 3.0210 / 3.1074 | 2.2298 / 2.2287 |
| RandomForest | деревья | 0.9896 | 0.9776 | 0.5230 / 1.9434 | 0.3627 / 1.2144 |
| CatBoost | градиентный бустинг | 0.9976 | 0.9947 | 0.3555 / 0.9126 | 0.2676 / 0.5705 |
| Stacking | финальный ансамбль | 0.9978 | 0.9948 | 0.3592 / 0.9055 | 0.2657 / 0.6056 |

### 5.3 Финальная модель

Финальный выбор: `Stacking`.
Причины:
- лучшая валидационная R²;
- стабильные тестовые показатели;
- сохранение простоты вывода благодаря мета-алгоритму `LinearRegression`.

### 5.4 Метрики по целевым переменным

Метрики на тестовой выборке для `Stacking`:
- Y1: `R² = 0.9988`, `RMSE = 0.3592`, `MAE = 0.2657`;
- Y2: `R² = 0.9909`, `RMSE = 0.9055`, `MAE = 0.6056`.

Метрики на валидационной выборке:
- Y1: `R² = 0.9990`, `RMSE = 0.2986`, `MAE = 0.2130`;
- Y2: `R² = 0.9965`, `RMSE = 0.5446`, `MAE = 0.4074`.

## 6. Архитектура решения и сервис

### 6.1 Архитектура пайплайна

Основные компоненты:
- `src/data/loader.py` — загрузка и сплит данных;
- `src/features/preprocess.py` — препроцессор (`StandardScaler` + `OneHotEncoder`);
- `src/models/registry.py` — загрузка модели по манифесту;
- `src/models/predictor.py` — логика предсказания, расчёт риска, рекомендации;
- `src/service/main.py` — FastAPI-сервис.

Пайплайн работает так:
1. API получает JSON с признаками `X1..X8`.
2. `CoolingPredictor` формирует DataFrame и делает предсказания для `Y1` и `Y2`.
3. Вычисляется уровень риска по порогам из `configs/serving.yaml`.
4. Генерируются рекомендации и топ-факторы.
5. API возвращает ответ в структуре `PredictionResponse`.

### 6.2 API и endpoints

Реализованные endpoints:
- `GET /health` — проверка доступности сервиса;
- `POST /predict` — прогнозирование нагрузок;
- `GET /info` — информация о загруженной модели.

Формат запроса `/predict`:
- `X1..X5` — float;
- `X6`, `X8` — int;
- `X7` — float.

Ответ `/predict` содержит:
- `heating_load`;
- `cooling_load`;
- `risk_level` (`low`, `medium`, `high`);
- `recommendations`;
- `top_factors`.

### 6.3 Технологический стек

- Python `>=3.14`;
- FastAPI для REST API;
- scikit-learn, CatBoost для моделей;
- pandas и numpy для данных;
- pydantic для валидации схем;
- pytest для тестов;
- Docker для контейнеризации.

## 7. Наблюдаемость, конфигурация и безопасность

### 7.1 Логи и наблюдаемость

- Сервис логирует события в `artifacts/logs/service.log`.
- Логи включают время ответа, результаты предсказаний и ошибки загрузки модели.
- Тренировочный лог сохраняется в `artifacts/logs/train.log`.

### 7.2 Конфигурация

Конфиги хранятся в `configs/`:
- `data.yaml` — параметры загрузки и описание колонок;
- `experiment.yaml` — конфигурации моделей и препроцессора;
- `serving.yaml` — параметры сервера и пороги риска.

Преимущество: изменение стратегии обучения или порогов риска не требует правки исходного кода.

### 7.3 Безопасность

- В репозитории нет секретных ключей и учетных данных.
- Конфигурационные файлы не содержат паролей.
- Данные — открытый CSV из UCI, без персональной информации.

## 8. Ограничения и дальнейшая работа

### 8.1 Текущие ограничения

- Набор данных небольшой, поэтому модель чувствительна к новым типам зданий.
- В сервисе нет механизма онлайн-мониторинга дрейфа данных.
- Пороговая логика риска фиксирована и базируется на эвристике.
- В `src/train.py` текущая реализация обучения является заглушкой, а фактическое обучение проводится в ноутбуках.

### 8.2 Дальнейшая работа
